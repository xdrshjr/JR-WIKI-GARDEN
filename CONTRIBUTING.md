## üìù Content Guidelines

### Quality Standards

- ‚úÖ **Accurate** - Information should be correct and up-to-date
- ‚úÖ **Clear** - Write in simple, understandable language
- ‚úÖ **Complete** - Include necessary context and examples
- ‚úÖ **Original** - Your own work, or properly attributed quotes

### File Organization

1. **Choose the right space** - Pick the appropriate top-level category
2. **Create subdirectories** - Group related content together
3. **Use descriptive names** - `YYYY-MM-DD-topic-name.md` format
4. **Add a README** - Each space/subspace should have a README.md

### Markdown Style

```markdown
# Document Title

## Overview

Brief description of what this document covers.

## Main Content

### Section 1

Content here...

### Section 2

More content...

## Summary

Key takeaways...

---

**Created**: YYYY-MM-DD  
**Updated**: YYYY-MM-DD  
**Tags**: tag1, tag2, tag3
```

## üîÑ Pull Request Process

### 1. Fork the Repository

Click the "Fork" button on GitHub to create your own copy.

### 2. Create a Branch

```bash
git checkout -b add/your-content-name
```

### 3. Add Your Content

- Place files in the appropriate space
- Follow the naming conventions
- Include a brief README update if needed

### 4. Commit Changes

```bash
git add .
git commit -m "Add: brief description of your content"
```

Commit message format:
- `Add: new tutorial on XYZ`
- `Update: correct outdated info in ABC`
- `Fix: typo in DEF`

### 5. Push and Create PR

```bash
git push origin add/your-content-name
```

Then create a Pull Request on GitHub with:
- Clear title
- Brief description of changes
- Any relevant notes

## ‚úÖ PR Checklist

Before submitting:

- [ ] Content is placed in the correct space
- [ ] File follows naming convention (`YYYY-MM-DD-title.md`)
- [ ] Markdown is properly formatted
- [ ] No sensitive information (passwords, API keys)
- [ ] Original content or properly attributed
- [ ] README updated if adding new subspace
